# Q5_0 -- 5-bit Block Quant

## Overview

| Property | Value |
|----------|-------|
| Full name | 5-bit Block Quantization |
| GGML type ID | 6 |
| Bits per weight | 5.5 bpw |
| Block size | 32 elements |
| Block bytes | 22 bytes |
| Compression ratio | 5.8x vs F32 |

## Data Layout

Each 22-byte block encodes 32 weights:

```
Offset  Size   Field    Description
------  -----  -------  -------------------------------------------
0       2      d        fp16 block scale
2       4      qh       uint32 high bits (1 per weight, 32 bits)
6       16     qs       16 bytes of 4-bit low nibbles
------  -----  -------  -------------------------------------------
Total:  22 bytes
```

### IMPORTANT: Split Nibble Layout

Q5_0 uses a **SPLIT** nibble layout, which is different from Q4_0's interleaved layout:

```
Elements  0-15:  LOW  nibbles of qs bytes 0-15,  qh bits 0-15
Elements 16-31:  HIGH nibbles of qs bytes 0-15,  qh bits 16-31
```

This means:
- Weight at position `j` (where `j < 16`): low nibble of `qs[j]`, high bit from `qh bit j`
- Weight at position `j` (where `j >= 16`): high nibble of `qs[j-16]`, high bit from `qh bit j`

**Do not confuse with Q4_0**, which interleaves even/odd positions within each byte (even=low nibble, odd=high nibble of the same byte index `j/2`).

## Dequantization Formula

```
if j < 16:
    low4 = qs[j] & 0x0F
else:
    low4 = (qs[j - 16] >> 4) & 0x0F

high_bit = (qh >> j) & 1
quant = low4 | (high_bit << 4)      // 5-bit value: 0-31

value = (quant - 16) * d            // centered around zero
```

The zero-point is fixed at 16, giving an effective signed range of -16 to +15.

## Bug History

**v1.4.0 and earlier:** The original Q5_0 implementation incorrectly assumed an interleaved nibble layout (like Q4_0), mapping even-indexed weights to low nibbles and odd-indexed weights to high nibbles within the same byte at position `j/2`. This produced incorrect dequantized values.

**v1.5.0:** Fixed to use the correct split layout where elements 0-15 read low nibbles of bytes 0-15, and elements 16-31 read high nibbles of bytes 0-15. The fix was verified bit-exact against llama.cpp reference output.

**Impact:** The bug caused **Gemma 3 models to produce garbage output**, because Gemma 3 Q4_K_M uses Q5_0 for Q, K, gate, and up projections. Since these are the most compute-intensive tensors in the forward pass, incorrect dequantization corrupted all downstream attention and FFN computations.

## Performance Characteristics

Q5_0 uses smaller 32-element blocks (vs 256 for Q5_K), which means more scale overhead per weight but simpler dequantization logic. The fixed zero-point of 16 (no per-sub-block minimum) makes Q5_0 slightly less flexible than Q5_K for representing weight distributions, but the dequantization is faster due to fewer lookups.

At the same 5.5 bpw as Q5_K, Q5_0 generally offers slightly lower quality because it lacks the per-sub-block scale/minimum granularity of K-quants. However, the simpler structure can be advantageous for certain tensor shapes.

## Typical Usage

- Gemma 3 models: Q/K/gate/up projections in Q4_K_M GGUF variants
- Some older GGUF files predating the K-quant series
- Mixed quantization where certain tensors benefit from the simpler block structure
