# Q8_0 -- 8-bit Block Quant

## Overview

| Property | Value |
|----------|-------|
| Full name | 8-bit Block Quantization |
| GGML type ID | 8 |
| Bits per weight | 8.5 bpw |
| Block size | 32 elements |
| Block bytes | 34 bytes |
| Compression ratio | 3.8x vs F32 |

## Data Layout

Each 34-byte block encodes 32 weights:

```
Offset  Size   Field    Description
------  -----  -------  -------------------------------------------
0       2      d        fp16 block scale
2       32     qs       32 x int8 quantized values
------  -----  -------  -------------------------------------------
Total:  34 bytes
```

Q8_0 is the simplest quantized format: one fp16 scale factor per block, and each weight is stored as a signed 8-bit integer. No nibble packing, no bit splitting, no lookup tables.

## Dequantization Formula

```
value = d * qs[i]      // qs[i] is signed int8 (-128 to +127)
```

## CUDA Kernel

| Property | Value |
|----------|-------|
| Kernel file | `matmul_q8_0.cu` |
| Strategy | Warp-per-row, `__ldg` texture cache reads |
| Alignment | 34 bytes -- NOT 4-byte aligned |

## SIMD Optimization

| Property | Value |
|----------|-------|
| Fused SIMD class | `SimdQ8_0FloatTensor` (java21) |
| Available since | v1.5.0 (dequant+dot), v1.6.0 (Q8_0 x Q8_0 integer dot), v1.12.0-dev (B2I lane-parallel rewrite) |
| CPU dot path | Fused SIMD dequant+dot via `ByteVector → B2I → I2F → FMA` (direct from mapped `MemorySegment`, zero scratch) |
| Special path | Q8_0 x Q8_0 integer dot product (no dequantization needed) |

### B2I lane-parallel rewrite (v1.12.0-dev, 2026-04-15)

Old SIMD kernel kept a scalar `for j in F_LEN` loop converting int8 → float via a `float[F_LEN]` scratch, then `FloatVector.fromArray` for the FMA — SIMD only in the final multiply. JFR flagged `SimdQ8_0FloatTensor.dot` as the #1 CPU hotspot (15391 samples on Qwen3-4B-Thinking Q8_0, 7× any other method). Rewritten to read 8 bytes as `ByteVector.fromMemorySegment`, widen via `convertShape(B2I, I_SPECIES)`, then `convertShape(I2F, F_SPECIES)` + FMA. Zero scratch allocation per dot call.

**Measured gains (Intel Core Ultra 7 155H CPU-only, apples-to-apples 30-token prompt):**
- **Qwen3-4B-Thinking Q8_0: 1.1 → 4.7 tok/s (+327%)**
- **Qwen3-1.7B Q8_0: 2.9 → 8.4 tok/s (+190%)**
- Qwen3-0.6B Q8_0: 6.6 → 12.2 tok/s (+85%)

### Q8_0 x Q8_0 Integer Dot Product

When both operands are Q8_0, the implementation uses a fast integer accumulation path:

```
for each aligned block pair:
    isum = sum(qs_a[i] * qs_b[i])  for i in 0..31    // integer multiply+accumulate
    result += d_a * d_b * isum                         // single float multiply at end
```

This avoids dequantizing either operand to float, using direct `MemorySegment` byte reads with integer accumulation. The float multiplication happens only once per block (for the scale product), not per element.

### Q4_0 x Q8_0 Mixed Dot Product

`Q4_0FloatTensor` also implements a fast integer path when the other operand is Q8_0, using the same integer accumulation strategy with inline nibble extraction.

## Performance Characteristics

Q8_0 is the highest quality standard quantization format. The 8-bit representation preserves nearly all model quality -- perplexity degradation vs F16/F32 is typically negligible.

The 34-byte block is not 4-byte aligned (34 = 2 * 17), which prevents vectorized loads in CUDA. However, the simple dequantization (single multiply per element) means compute overhead is minimal compared to complex K-quant formats.

At 8.5 bpw, model sizes are roughly double those of Q4_K:
- 7B model: ~7.3 GB
- 13B model: ~13.8 GB

The extra 0.5 bpw overhead (8.5 vs 8.0) comes from the 2-byte fp16 scale per 32-element block.

## Typical Usage

- DeepSeek-Coder-V2-Lite Q8_0
- High-quality inference where memory is not the primary constraint
- Activation quantization for KV cache in some implementations
- Reference/validation quantization for comparing against lower-bit formats
- When model quality is critical and the 2x size penalty vs Q4_K is acceptable
