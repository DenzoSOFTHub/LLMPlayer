# Q5_1 -- 5-bit Block Quant with Min

## Overview

| Property | Value |
|----------|-------|
| Full name | 5-bit Block Quantization with Asymmetric Min |
| GGML type ID | 7 |
| Bits per weight | 6.0 bpw |
| Block size | 32 elements |
| Block bytes | 24 bytes |
| Compression ratio | 5.3x vs F32 |

## Data Layout

Each 24-byte block encodes 32 weights:

```
Offset  Size   Field    Description
------  -----  -------  -------------------------------------------
0       2      d        fp16 block scale
2       2      m        fp16 block minimum (zero-point offset)
4       4      qh       uint32 high bits (1 per weight, 32 bits)
8       16     qs       16 bytes of 4-bit low nibbles
------  -----  -------  -------------------------------------------
Total:  24 bytes
```

The block size of 24 IS divisible by 4, so all multi-byte reads (uint32 for `d+m`, uint32 for `qh`, uint32 chunks of `qs`) are 4-byte aligned and can use vectorized loads on GPU.

### Split Nibble Layout

Q5_1 uses the same **SPLIT** nibble layout as Q5_0 (different from Q4_0/Q4_1 interleaved layouts):

```
Elements  0-15:  LOW  nibbles of qs bytes 0-15,  qh bits 0-15
Elements 16-31:  HIGH nibbles of qs bytes 0-15,  qh bits 16-31
```

This means:
- Weight at position `j` (where `j < 16`): low nibble of `qs[j]`, high bit from `qh bit j`
- Weight at position `j` (where `j >= 16`): high nibble of `qs[j-16]`, high bit from `qh bit (j+16)`

## Dequantization Formula

```
if j < 16:
    low4 = qs[j] & 0x0F
else:
    low4 = (qs[j - 16] >> 4) & 0x0F

high_bit = (qh >> j) & 1
quant = low4 | (high_bit << 4)      // 5-bit value: 0-31

value = quant * d + m                // asymmetric, no zero centering
```

Unlike Q5_0 which centers at 16 (`(quant - 16) * d`), Q5_1 stores an explicit per-block minimum `m`, allowing the encoder to fit asymmetric weight distributions more accurately. This costs 2 extra bytes per block (24 vs 22) but typically improves quality, especially for tensors with non-zero-centered distributions.

## CUDA Kernel

| Property | Value |
|----------|-------|
| Kernel file | `matmul_q5_1.cu` |
| Strategy | Warp-per-row |
| Alignment | 24 bytes IS 4-byte aligned -- safe for vectorized `uint32` loads |
| Available since | v1.10.0 |

The kernel reads `d+m` as a single `uint32`, `qh` as a `uint32`, and the 16 packed nibble bytes as four `uint32` chunks for efficient memory access.

## CPU Implementation

| Property | Value |
|----------|-------|
| Class | `Q5_1FloatTensor` |
| SIMD class | not yet available (scalar fallback) |

The CPU dot product accumulates `scale * dot(quant, input) + min * sum(input)` per block.

## Performance Characteristics

Q5_1 provides slightly better quality than Q5_0 at 9% larger memory footprint (24 vs 22 bytes per block). The key advantage is asymmetric quantization via the explicit minimum field, which reduces dequantization error for tensors with biased weight distributions.

The 24-byte aligned layout makes the GPU kernel slightly faster than Q5_0 (which has 22-byte non-aligned blocks requiring byte-by-byte assembly of multi-byte fields).

At 6.0 bpw, Q5_1 sits between Q5_0 (5.5 bpw) and Q6_K (6.5 bpw) in the quality/size trade-off space.

## Typical Usage

- Older GGUF files predating the K-quant series that needed asymmetric quantization
- Some GGML legacy models that used Q5_1 for better quality on biased tensors
- Mostly superseded by Q5_K, which provides per-sub-block scales and minimums for even finer-grained quantization within a 256-element super-block
