# F16 -- IEEE Float 16 (Half Precision)

## Overview

| Property | Value |
|----------|-------|
| Full name | IEEE 754 Half-Precision Floating Point |
| GGML type ID | 1 |
| Bits per weight | 16 bpw |
| Block size | 1 element |
| Block bytes | 2 bytes |
| Compression ratio | 2x vs F32 |

## Data Layout

Each weight is stored as a single 16-bit IEEE 754 half-precision float:

```
Bit layout (16 bits):
[sign (1 bit)] [exponent (5 bits)] [mantissa (10 bits)]
```

F16 has a 5-bit exponent (vs BF16's 8-bit), giving a smaller dynamic range but higher mantissa precision (10 bits vs 7 bits).

## Dequantization Formula

F16 to F32 conversion requires manual bit manipulation because Java does not have native half-float support. The conversion is handled by `Float16.toFloat()`:

```
Input: 16-bit fp16 value
  sign     = (bits >> 15) & 1
  exponent = (bits >> 10) & 0x1F
  mantissa = bits & 0x3FF

Conversion cases:
  exponent == 0:  subnormal → value = (-1)^sign * 2^(-14) * (mantissa / 1024)
  exponent == 31: infinity/NaN
  otherwise:      value = (-1)^sign * 2^(exponent - 15) * (1 + mantissa / 1024)
```

This is more expensive than BF16's simple bit shift but still fast compared to quantized format dequantization.

## Performance Characteristics

F16 vs BF16 trade-offs:

| Property | F16 | BF16 |
|----------|-----|------|
| Exponent bits | 5 | 8 |
| Mantissa bits | 10 | 7 |
| Max value | ~65504 | ~3.39e38 |
| Precision | ~3 decimal digits | ~2 decimal digits |
| Dequant cost | Bit manipulation | Single shift |

F16 has higher precision per value (10-bit mantissa) but a much smaller dynamic range (max ~65504). For LLM weights, which are typically small values near zero, both F16 and BF16 work well. However, BF16 is more robust because its full F32 exponent range prevents overflow in edge cases.

The dequantization cost of F16 is slightly higher than BF16 due to the manual bit manipulation, but both are much cheaper than any quantized format's dequantization.

A 7B model in F16 uses approximately 14 GB.

## Typical Usage

- qwen2.5-coder-14b-instruct-fp16 (as used in LLMPlayer)
- Models originally distributed in fp16 format
- When hardware has native fp16 support and BF16 is not available
- Slightly higher precision than BF16 when values are in F16's dynamic range
