# BF16 -- Brain Float 16

## Overview

| Property | Value |
|----------|-------|
| Full name | Brain Floating Point 16 (bfloat16) |
| GGML type ID | 30 |
| Bits per weight | 16 bpw |
| Block size | 1 element |
| Block bytes | 2 bytes |
| Compression ratio | 2x vs F32 |

## Data Layout

Each weight is stored as a single 16-bit bfloat16 value:

```
Bit layout (16 bits):
[sign (1 bit)] [exponent (8 bits)] [mantissa (7 bits)]
```

BF16 has the same exponent range as F32 (8-bit exponent) but a truncated mantissa (7 bits vs 23 bits). This preserves the dynamic range of F32 while halving the storage.

## Dequantization Formula

BF16 to F32 conversion is a simple bit shift -- the BF16 value occupies the upper 16 bits of a 32-bit float:

```java
float value = Float.intBitsToFloat(((int) bf16_bits) << 16);
```

This is the cheapest possible dequantization: a single integer left-shift followed by a bit reinterpretation. No arithmetic operations needed.

## CUDA Kernel

| Property | Value |
|----------|-------|
| Kernel file | `matmul_bf16.cu` |
| Strategy | Warp-per-row, `__ldg` with shift |
| Available since | v1.6.0 |

The kernel reads 16-bit values, shifts left by 16, and reinterprets as float directly on the GPU.

## SIMD Optimization

| Property | Value |
|----------|-------|
| Fused SIMD class | None |
| CPU dot path | Uses base `FloatTensor.dot()` with `getFloat()` dequantization |

No fused SIMD variant exists because the dequantization is trivial (single shift). The base implementation calling `getFloat()` per element is already efficient for BF16.

## Performance Characteristics

BF16 provides near-lossless quality at 2x compression vs F32. The mantissa truncation (23 bits to 7 bits) introduces small rounding errors but preserves the full exponent range, meaning:

- No overflow/underflow issues compared to F16 (which has only 5 exponent bits)
- Maximum representable value: ~3.39e38 (same as F32)
- Minimum subnormal: ~1.18e-38 (same as F32)
- Precision: ~2 decimal digits (vs ~7 for F32)

For LLM inference, BF16 quality is essentially indistinguishable from F32. The primary use case is when maximum quality is desired and memory allows the 2x overhead vs quantized formats.

A 7B model in BF16 uses approximately 14 GB -- too large for most consumer GPUs but viable for systems with 16+ GB VRAM or sufficient RAM for CPU inference.

## Typical Usage

- High-quality model variants where quantization artifacts are unacceptable
- Training and fine-tuning (BF16 is the default training precision for many LLMs)
- Reference comparisons against quantized versions
